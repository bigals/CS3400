Browser Compatability Testing

Browserling

Browserling requires a URL to a website to be applied, and once entered, you can test the full interface of your site on multiple browsers for a set period of time. Once the time has expired, jusr re-enter the URL of your site, and you can continue testing compability.

Supported Browsers:
  • IE9
  • Chrome(1.0-37.0)
  • Firefox(3.0-32.0)
  • Safari(4.0-5.1)
  • Opera(10.0-24.0)

Spoon

Spoon runs VMs in your browser in order to allow you to test your web site on most major browsers, including the Opera Mobile and Firefox mobile browsers.

Supported Browsers:
  • IE(6-11)
  • Chrome(11-37)
  • Firefox(2-32)
  • Safari(3-5)
  • Opera(9-20)
  • Firefox Mobile(2, 5)
  • Opera Mobile(6, 11.5, 12)

BrowserStack

BrowserStack runs VMs in browser to test almost all browsers, including various mobile device resolutions as well. BrowserStack is a paid monthly subscription service, but does have a Free Trial process, which doesnt involve credit card information of any kind. BrowserStack could be usefull in testing certain browsers that are not supported by Browserling or Spoon.

Supported Browsers:
  • IE(8-11)
  • Chrome(14-39)
  • Firefox(3-34)
  • Safari(4.0-5.1)
  • Opera(10.6-25.0)
  • iOS(3.0-7.0)
  • Android(1.5-4.1)

Chrome Dev Tools

Chrome dev tools, in the latest version of chrome, allows you to emmulate various mobile devices, and change the resolution of the screen itself to allow for testing of the scaling of your webpage. In addition Dev tools allows you to detect memory leaks, and framerates of renderings on your webpage.

Supported Browsers:
  • Chrome

Version Control

Git via GitHub

Git is a version control service, which was loosely modeled off of BitKeeper, the original version control system used to maintain the Linux Kernel. Git relies on checksums to ensure that it knows about every change to files or directories in the repository.

Creating/Cloning a repository

"git init" will start the current directory being tracked with git.

"git clone (URL of hosted git repository)" will clone the repository at the specified URL, therefore creating a local directory with the same name, as the hosted repository.

States/Staging/Commiting Files

Checking the status, to see which files are in which of the three state (i.e. untracked, unmodified, modified, and staged) is done through the "git status" command. When the status shows there are files that are in the untracked state, the next step would be to use the "git add (file or path)" command. The (file or path) parameter to the "git add" command will take a filename to be tracked, or a directory. In the later case, Git will begin tracking every file/directory within the specified directory.

The "git add" command is also used to stage files that have had changes made to them. So after changes are made to a file, and before you are able to commit the changes, you must run the "git add" command to stage the modified files.

(Note: "git add ." would begin tracking any file or directory not currently tracked within the current directory, or stage any file or directory that has been modified, but not yet staged.)

"git commit -m "(message explaining commit here)"" will commit all of the staged changes and apply the message that is specified within the double quotes to all of the staged changes that are being commited.

Differences

"git diff" will show modifications which have not yet been staged.

"git diff --cached" or "git diff --staged" will show changes that have been staged, and that will go into your next commit. In other words, it will compare your staged chnages to your last commit.

Removing/Moving Files

"git rm (filename here)" will remove the file specified by the (filename here) parameter.

"git mv (file from) (file to)" will move the file specified by the (file from) parameter to the file specified by the (file to) parameter.

The Commit Log

"git log [-p] [-2] [--stat] [--pretty=format:"(format string)"]" will display the commit history for the project, whos current directory the command is run from. The optional [-p] parameter will include the diff along with each commit, and the optional [-2] parameter will limit the log output to the last 2 commit entries.

The optional [--stat] parameter will print, below each commit entry the modified file names, how many files were changes, and how many lines in the files were added, and removed.

The optional [--pretty=format:"(format string)"] parameter alows the output of the log for each commit entry to be completly user specifed, where the (format string) parameter is a "-delimited string using the following escape characters:

Option Description of Output
%H Commit hash
%h Abbreviated commit hash
%T Tree hash
%t Abbreviated tree hash
%P Parent hashes
%p Abbreviated parent hashes
%an Author name
%ae Author e-mail
%ad Author date (format respects the --date= option)
%ar Author date, relative
%cn Committer name
%ce Committer email
%cd Committer date
%cr Committer date, relative
%s Subject

Reverting/Discarding Chnages

"git commit --ammend" will replace the previous commit's actions with the current commit's actions.

"git reset HEAD (filename)" will unstage the file, specified by the (filename) parameter.

"git checkout -- (filename)" will discard any chnages that were made to the file, specified by the (filename) parameter.

Push/Pull

"git pull" will automatically fetch, and merge from the remote (i.e. server) to your local working copy.

"git push" will push your local copy upstream to the remote (i.e. server). If however, someone has already pushed to the remote, after you had last pulled, then the command will fail, and your push will be rejected. To fix this, a "git pull" command should be run, and then the push command should succeed.

A Simple Workflow for using Git Effectivly

To use Git effectivly, a series of commands can be run at different points during the time you spend working on a team project:

  1. First, always pull from the remote before you start working using the "git pull" command.
  2. Next, whenever you have made a series of changes you are happy with to a particular file/files, immediatly stage the changes using the "git add" command.
  3. After the modifications/new files have been staged, you have to commit the changes, using the "git commit -m "(comment)"" command. Always supply a comment along with the commit, in order to create an easily distinguishable history of commits, in case something goes wrong.
  4. Lastly when you have finished a substantial amount of work, that is correct, and you are happy with, push the changes upstream to the remote using the "git push" command.
